Implementation: Define a Performance Metric¶

It is difficult to measure the quality of a given model without quantifying its performance over training and testing. This is typically done using some type of performance metric, whether it is through calculating some type of error, the goodness of fit, or some other useful measurement. For this project, you will be calculating the coefficient of determination, R2, to quantify your model's performance. The coefficient of determination for a model is a useful statistic in regression analysis, as it often describes how "good" that model is at making predictions.

The values for R2 range from 0 to 1, which captures the percentage of squared correlation between the predicted and actual values of the target variable. A model with an R2 of 0 is no better than a model that always predicts the mean of the target variable, whereas a model with an R2 of 1 perfectly predicts the target variable. Any value between 0 and 1 indicates what percentage of the target variable, using this model, can be explained by the features. A model can be given a negative R2 as well, which indicates that the model is arbitrarily worse than one that always predicts the mean of the target variable.

Question 3 - Training and Testing¶

What is the benefit to splitting a dataset into some ratio of training and testing subsets for a learning algorithm?
Hint: What could go wrong with not having a way to test your model?

Answer: The training data is used to fit the model in the first place, and the testing data used to check how accurate it works. If all the data was used to train the model then we can't get a fair measure of how well it works since we would be testing the accuracy using data the model has already seen. Thus we need to hold some out as testing data to confirm its performance.

Furthermore, if we really used all the data for training and had no way to test the model we would be left with a dangerous risk of overfitting. The model might be made arbitrally complex to fit all the training data without capturing the actual underlying pattern. Holding back some data for training allows us to check performance and ensure this overfitting is not happening.

Question 7 - Grid Search¶

What is the grid search technique and how it can be applied to optimize a learning algorithm?

Answer: The machine learning algorithms we use have different hyperparameters which control different aspects over exactly how they work. A good example is the max_depth parameter for the Decision Tree algorithm that we used so far in this project which sets an upper bound on the number of levels of hierachy in the tree.

When we want to apply a learning algorithm to a problem we have to set these hyperparameters, but it wont usually be clear exactly what the optimal parameters should be. It can be complicated enough just trying to tune one parameter as we saw so far in this project, but in reality we might want to set 2,3 or even more different parameters. Furthemore changing one parameter will almost certainly change the optimal value for one of the other ones. As the number of hyperparameters increases, the number of options increases exponentially, so we need a systematic way to keep track of and determine the best parameters to use. This is where the grid search technique comes in.

In the grid search technique we tell the algorithm which n parameters we want to tune and a list of potential values for each. It makes a n-dimensional grid of all possible combinations of all parameters and then runs our target machine learning algorithm with each of those combinations, keeping track of the scores/errors obtained by each. It then selects the parameters which created the highest score/lowest error and presents these as the optimal tuning values for the machine learning algorithm.

Question 8 - Cross-Validation¶

What is the k-fold cross-validation training technique? What benefit does this technique provide for grid search when optimizing a model?
Hint: Much like the reasoning behind having a testing set, what could go wrong with using grid search without a cross-validated set?

Answer: As already discussed it is important for us to split the data into training and testing sets. But when we do grid search we need further subdivide the data into training, validation and testing sets. The model is first created with the training set, then hyperparameters are tuned with the validation set and finally the models score evaluated with the testing set. But we have a difficult trade-off to make in determining how much data to allocate to each sub-set. The more data we use for training in theory the better the model we can make. But the more data we use for validation the better we can choose the hyperparameters. And finally the more data used for testing, the better we can evaluate how well the model actually worked. Splitting up the data into too small groups can lead to bad performance and this is especially important if we have only have a small data set to start off with.

Using k-fold cross-validation training technique can help avoid us having to make this difficult decision. We still have to separately set aside a sub-portion of the data for testing, but then k-folds helps us maximise the use of the remaining data to do both training and validation. K-folds splits the remaining data into k equally-sized subsets, or 'folds'. It then trains the model using k-1 of the folds and uses the remaining fold for the validation. It then repeats this process k times, each time using a different fold for the validation and the remaining folds for training. The final reported performance of the model is the average score across all k runs. This increases computational complexity, but means all data is used at some point in both training and validation so maximises both the potential of creating the best possible model and having best hyperparameters. It avoids wasting data which is especially important when the number of samples is small.

If we did grid-search without having a cross-validated set we could end up overfitting the hyperparameters to the testing data rather than correctly modelling the underlying trends.

# TODO: Import 'make_scorer', 'DecisionTreeRegressor', and 'GridSearchCV'
from sklearn.metrics import make_scorer
from sklearn.tree import DecisionTreeRegressor
from sklearn.grid_search import GridSearchCV

def fit_model(X, y):
    """ Performs grid search over the 'max_depth' parameter for a 
        decision tree regressor trained on the input data [X, y]. """
    
    # Create cross-validation sets from the training data
    cv_sets = ShuffleSplit(X.shape[0], n_iter = 10, test_size = 0.20, random_state = 0)

    # TODO: Create a decision tree regressor object
    regressor = DecisionTreeRegressor()

    # TODO: Create a dictionary for the parameter 'max_depth' with a range from 1 to 10
    params = {'max_depth':(range(1,11))}

    # TODO: Transform 'performance_metric' into a scoring function using 'make_scorer' 
    scoring_fnc = make_scorer(performance_metric)

    # TODO: Create the grid search object
    grid = GridSearchCV(estimator = regressor, param_grid = params, scoring = scoring_fnc, cv = cv_sets)

    # Fit the grid search object to the data to compute the optimal model
    grid = grid.fit(X, y)

    # Return the optimal model after fitting the data
    return grid.best_estimator_
